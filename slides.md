# Documentation as Code  
### Proposition d’innovation — Industrialisation documentaire

---

## 1) Contexte & problème actuel

- volumétrie documentaire croissante
- co-rédaction simultanée entre plusieurs métiers
- contrôles / remarques / approbations successives
- pertes de temps : consolidation de versions, email ping-pong
- traçabilité insuffisante (qui modifie quoi, quand, et pourquoi)

> Aujourd’hui la documentation **se comporte comme du code**, mais nos outils ne suivent pas.

---

## 2) Positionnement stratégique

| Enjeu entreprise | Apport de l’innovation |
|---|---|
| simplifier les processus | workflow rationalisé |
| digitaliser les pratiques | passage au “doc as code” |
| absorber la croissance | méthode scalable |
| sécuriser la qualité | versionnement structuré |

> C’est une brique d’industrialisation documentaire.

---

## 3) L’idée — Documentation As Code

Principe : la documentation est traitée comme du code source :

- stockage dans dépôt Git
- versionning natif
- review en Pull Request
- validation par merge
- historique complet automatique

Stack inspirée de : GitHub / GitLab / Gitea

---

## 4) Fonctionnement (schéma conceptuel)

Rédacteur → commit → pull request → revue → approbation → merge → version publiée

- commentaires contextualisés sur diff
- historique infalsifiable
- preuve d’approbation intégrée

> La mécanique est déjà éprouvée dans le monde logiciel.

---

## 5) Bénéfices attendus

- gain de productivité
- suppression des consolidations manuelles
- revue plus rigoureuse
- auditabilité et traçabilité automatique
- montée en charge sans surcharge administrative

---

## 6) Risques / points d’attention

- adoption / conduite du changement
- format textuel (Markdown / AsciiDoc)
- formation aux workflows Git
- gouvernance / droits / sécurité à cadrer

---

## 7) Plan d’expérimentation (PoC 3 mois)

| Phase | Durée | Contenu |
|---|---:|---|
| cadrage | 1 sem | choix techno, périmètre, gouvernance |
| formation pilote | 1–2 sem | 8–12 personnes |
| expérimentation | 6 sem | vrai projet réel |
| bilan | 2 sem | retours, KPIs |

KPIs : temps de cycle / satisfaction / taux d’adoption / 0 commentaires perdus

---

## 8) Besoin décision CODIR

Valider :

- l’ouverture du chantier Doc As Code
- le lancement d’un PoC cadré
- allocation minimale :
  - 1 tech lead
  - 1 product owner
  - 1 projet pilote
  - 8–12 utilisateurs

> Objectif : **démontrer le gain avant industrialisation**.
